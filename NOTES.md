## 待办 / 想法

- [ ] **上云 · 账号与权限（已定方向，未实施）**
  - 目标：**仅登录用户可编辑**；若干**协作者**账号（小团队，非公众投稿）
  - 访客：**只读**浏览地图 / 影像馆 / 档案馆 / 语录
  - 第一期（当前）：**暂不接登录**；官方数据仍通过 **改代码内置数据 + push 部署** 维护；协作者共用 GitHub 协作
  - 第二期：API + 库 + 登录；图片迁对象存储；前端 `useMarkers` 等改读写云端
  - 登录方式：**邮箱**（邮箱 + 密码或魔法链接；协作者账号由你邀请/登记）
  - 权限模型参考：**公开只读 + 已登录协作者可编辑**（非「人人可改 localStorage」）
- [ ] **导览 Agent（已定方向，未实施）**
  - **定位**：站内 AI 助手，辅助访客/协作者理解地图内容；**不改动现有 markers / gallery / quotes / archives 数据结构**
  - **三类能力（分阶段）**：
    1. **检索导览（第一期）** — 只读
       - 按自然语言搜索地点、语录、档案馆、影像馆
       - 组织回答，并可触发地图跳转、选中标记、打开详情等 UI 操作
       - 查不到时明确说「暂无记录」，不编造历史
    2. **调用站内功能改数据（第二期）** — 需登录 + 协作者权限
       - 代劳添加/编辑/删除地点，上传/删改图片等
       - 底层仍走现有 `addMarker` / `updateMarker` / `useGallery` 等逻辑，不另建数据模型
       - 建议「Agent 生成草稿 → 人工确认 → 再写入」，避免误操作
       - 依赖 **上云 · 账号与权限** 完成后才可对公众安全开放
    3. **风格化寄语（第三期）** — 资料丰富后
       - 在事实准确、可溯源的前提下，参考真实语录/档案表述习惯，以该人物文献语气鼓励后生
       - 与「检索导览」分离为独立模式（如「导览模式」vs「寄语模式」）；寄语模式约束更严、更偏引用原文
  - **技术方案（概要）**：
    - 前端：新增 `AgentPanel` 聊天面板（交互模式对齐 `QuotesPanel` / `GalleryPanel`）；Header 加入口
    - 后端：Vercel Serverless `api/chat.js` + Vercel AI SDK；**API Key 仅存服务端环境变量**
    - Agent 通过 **tool calling** 调用工具：`searchMarkers`、`getMarkerDetail`、`searchQuotes`、`searchArchives`、`searchGallery`、`navigateMap` 等；第二期再加写操作工具
    - **不改** `localStorage` 键名、marker 字段 schema、现有 hooks 接口（第一期为只读消费者）
  - **模型与费用**：
    - 试水：**Google Gemini Flash 免费层**；国内长期：**DeepSeek**（注册赠送 + 后续按量，成本极低）
    - 额度按 **开发者 API 账号** 计费/限流，全站访客共享同一 Key，非每人独立配额
    - 非商业、自费可承受；需在 `api/chat.js` 自加 **IP / 全站日限流** 防刷
  - **与登录的关系**：网站登录管「谁能改数据」；LLM API Key 管「AI 费用算在谁头上」——两者独立。第一期导览可不要求登录
  - **第一期 MVP 清单**：
    - [ ] `api/chat.js` + 环境变量配置
    - [ ] `AgentPanel.jsx` + Header 按钮
    - [ ] 工具：`searchMarkers` + `navigateMap`（最小闭环）
    - [ ] System prompt + 内置官方数据注入服务端（暂不同步访客 localStorage）
    - [ ] 简单限流（IP 或全站日上限）
  - **2026-05-31** — 与助手确认 Agent 三阶段方向、架构与不碰数据结构原则
